# Pokémon Collection V3 — Decision Log

This living document records important architectural and product decisions and, crucially, why they were made.

| Date | Phase | Decision | Reason and lasting effect |
|---|---|---|---|
| 2026-07 | Project restart | Build Pokémon Collection V3 as a clean React, TypeScript and Supabase project instead of continuing the previous patched application. | The previous application had accumulated overlapping patches, regressions and technical debt. V3 prioritizes long-term maintainability and stable evolution. |
| 2026-07 | Workflow | Keep `main` stable and use one branch and PR per clear objective. | Small isolated changes are easier to review, test, revert and reason about. |
| 2026-07 | Workflow | Require explicit technical review and UX review before merging user-facing work. | Functional correctness alone did not produce a product-ready result during the set-card work. Both data safety and user experience must be approved. |
| 2026-07 | Workflow | Allow multiple correction rounds inside the same PR while its original purpose remains unchanged. | Local Codex makes iterative review practical. Small corrections should not fragment one coherent phase into unnecessary PRs, but a newly emerging goal must become a separate phase. |
| 2026-07 | Architecture | Evaluate long-term architecture and reuse before implementation. | Early assessment prevents page-specific duplication and reduces later technical debt. Architecture precedes coding, especially for large product areas. |
| 2026-07 | Core model | Use `cards_catalog` for card metadata and `collection_cards` for collection-specific ownership and status. | It prevents per-user metadata duplication and supports multiple profiles, collections, wishlist and trade using one catalog. |
| 2026-07 | Legacy | Treat `public.cards` as legacy and exclude it from new runtime functionality. | New functionality must use the normalized V3 model and must not recreate dependencies on historical structures. |
| 2026-07 | Sets | Use `sets_catalog` as the canonical source for set metadata and project `set_code` as the internal identifier. | Consistent set identity is required for progress, filtering, catalog synchronization and future set browsing. Internal codes remain hidden from end users. |
| 2026-07 | Phase 5V | Keep intelligent collection filter options in a database function. | Set and rarity options must reflect the active collection and each other without loading all cards into the browser. Search text intentionally does not alter those option lists. |
| 2026-07 | Phase 5W | Apply only proven set-code mappings and reject uncertain automatic mappings. | Incorrect set identity would contaminate progress and future imports. Controlled mappings linked 287 cards with zero remaining incorrect targets in the verified scope. |
| 2026-07 | Phase 5Z | Interpret the 2,190-row imported collection as a duplicated Dex-import structure containing placeholder and enriched versions. | Read-only analysis showed a systematic overlap rather than a genuine 2,190-card collection. This justified a guarded cleanup instead of treating every row as unique ownership. |
| 2026-07 | Phase 6A | Reduce Lars' collection from 2,190 rows to a verified baseline of 1,095 unique catalog links. | Placeholder links were replaced by enriched links where proven safe, preserving real ownership while removing the duplicate import structure. |
| 2026-07 | Phase 6A | During the one-time repair, reduce imported duplicate quantities of 2 to 1. | The user confirmed these quantities came from the faulty doubled import. This is not a general future rule for genuine duplicate physical cards. |
| 2026-07 | Phase 6B | Preserve 15 unresolved placeholders. | They represent real cards in Lars' collection but have no proven unique enriched target. Automatic cleanup or deletion is therefore forbidden. |
| 2026-07 | Collection status | Keep `owned`, `wishlist`, `trade` and `missing` in `collection_cards`. | These are collection-specific states of the same catalog card. A separate wishlist catalog would duplicate identity and complicate future transitions. |
| 2026-07 | Phase 7A | Permit narrowly constrained inserts into a user's own `collection_cards`. | The ownership path is authenticated user → profile → collection. Initial inserts are limited to quantity 1, Near Mint and owned until later UI and policy phases add broader editing. |
| 2026-07 | PR 89 | Close the initial Collection-page add-card PR without merge. | The implementation was technically functional, but the product flow was wrong and the catalog was nearly identical to Lars' collection. Correct add flows start from an opened set or a global full-catalog search. |
| 2026-07 | Phase 7B | Make Supabase `cards_catalog` the runtime catalog and use external APIs only for import and synchronization. | Direct external runtime search would expose the user experience to latency, outages and rate limits and would make owned and wishlist joins inefficient. |
| 2026-07 | Phase 7B | Support multiple external sources through `card_external_references`. | Stable internal card IDs must survive source changes. One card can carry legacy, Pokémon TCG API and future source references without duplicating the internal record. |
| 2026-07 | Phase 7B-1A | Remove exactly 1,095 unused legacy placeholders from `cards_catalog`. | After collection cleanup, these records had no collection links, no images and no market data. Removing them created a clean 1,095-record baseline without touching active cards. |
| 2026-07 | Phase 7B-1B | Backfill 1,058 proven Pokémon TCG API references without replacing internal IDs. | Existing URL data allowed exact source-ID derivation. The backfill improves future idempotent matching while preserving all collection links. |
| 2026-07 | Phase 7B-1C | Add trigram and set/number indexes before expanding the catalog. | Global search and set pages must remain server-side, paginated and fast as the catalog grows to thousands of cards. |
| 2026-07 | Import strategy | Import and synchronize in small resumable batches, preferably one set per run. | A full catalog operation in one serverless request is difficult to retry, validate and recover. Per-set batches isolate failures and allow expected/received checks. |
| 2026-07 | Import safety | Prohibit automatic catalog deletes and changes to collection quantity, condition or status during synchronization. | External sources can be temporarily incomplete or inconsistent. Source instability must never cause user-data loss. |
| 2026-07 | PR 91 | Close the read-only Vercel Function proof of concept without merge. | The proof of concept confirmed that the Vercel integration and Pokémon TCG API connectivity work, but live Function calls are not robust enough for bulk import because of timeout and recoverability limits. |
| 2026-07 | Phase 7B-2B | Use a controlled local Node/TypeScript script for catalog import instead of a live Vercel Function request. | A local operator-controlled process supports longer execution, explicit retries, per-set recovery, deterministic validation and safer write authorization. |
| 2026-07 | Phase 7B-2B | Make dry-run the permanent default and require an explicit `--write` option for every database-changing run. | Accidental writes must be impossible during normal inspection. Operators must first review expected counts, matches, conflicts and planned mutations before explicitly enabling writes. |
| 2026-07 | Phase 7B-2B | Require catalog imports to be fully idempotent and match primarily through `card_external_references`. | Repeated runs must preserve stable `cards_catalog.id` values, avoid duplicate cards and references, and never modify `collection_cards`. Source plus external card ID is the primary identity path; ambiguous matches block writes. |
| 2026-07 | Phase 7B-2E | Allow catalog writes only after a newly executed green pre-write validation, with dry-run as the default and exact `--write` as explicit authorization. | New cards and references are written in small idempotent, recoverable batches; existing metadata is never overwritten automatically, and `collection_cards` remains entirely outside the catalog import process. |
| 2026-07 | Phase 7C-1B | Runtime queries that can span more than one Supabase/PostgREST response page must be explicitly paginated or server-side aggregated; set completion counts unique `card_catalog_id` values with a physical-possession status. | Supabase/PostgREST response limits must never silently create incomplete results. `owned` and `trade` count as collected, `wishlist` and `missing` do not, and quantity does not increase completion. |
| 2026-07 | Phase 7B-2F1 | Read-only dry-runs may generically analyse multiple valid set IDs; writes remain explicitly authorized per set in code; dry-run access never grants implicit write authority. | Candidate sets can be analysed through the existing metadata, matching and theoretical write-plan flow without adding database write capability. The proven `sv3pt5` write path remains the only authorized path until a separately reviewed phase changes the allowlist. |
| 2026-07 | Phase 7B-2F2 | Authorize `sv3` (`Obsidian Flames`) as a second explicit catalog write set while preserving the existing guarded importer flow. | The read-only pilot passed with 230/230 cards, reliable setmapping, zero ambiguous/conflict/blocked items and 348 theoretical writes. The allowlist remains per-set so future catalog expansion still requires separate review and approval. |
| 2026-07 | Phase 7B-2F3 | Add a controlled batch runner that orchestrates the existing single-set importer instead of duplicating import logic. | Manual set-by-set importing is too time-consuming, but the proven validation, write and post-check behavior must remain centralized. The batch runner executes sets sequentially, validates importer output per step and stops on failed safety checks. |
| 2026-07 | Phase 7B-2F4 | Gebruik de lokale `PokemonTCG/pokemon-tcg-data` JSON-structuur eerst alleen read-only en behoud `pokemon_tcg_api` als externe identity source voor deze mirror. | De dataset heeft dezelfde set- en kaart-ID-structuur als de bewezen API-bron, waardoor bestaande external references en stabiele interne IDs opnieuw kunnen worden gevalideerd zonder API-timeouts. Writeautorisatie blijft een aparte beslissing totdat de bronversie en volledige setvalidatie zijn gecontroleerd. |
| 2026-07 | Phase 7B-2F4 correction | Gebruik `--set` als setcontext voor upstream per-set JSON-bestanden, omdat `cards/en/{set}.json` niet op ieder kaartitem een `set`-object herhaalt. | De bron wordt per set aangeleverd; optionele `set.id`-metadata blijft een controlepunt, maar het ontbreken ervan is geen fout. Hierdoor sluit de parser aan op de werkelijke bronvorm zonder de read-only veiligheidsgrens te wijzigen. |
| 2026-07 | Phase 7B-2F5 | Bewaar stabiele uitgebreide kaartdetails als optionele JSONB in `cards_catalog` en toon alleen beschikbare waarden in het gedeelde Card Detail. | Geneste API-velden zoals attacks, abilities en legalities passen niet veilig in losse kernkolommen. JSONB houdt de catalogus uitbreidbaar; de UI blijft fail-safe en `collection_cards` blijft volledig gescheiden. |
| 2026-07 | Phase 7B-2F6 | Gebruik een aparte gecontroleerde detail-backfillrunner met dry-run als standaard en alleen lege `card_details` als update-doel. | Bestaande catalogusrecords moeten verrijkt kunnen worden zonder kernmetadata te overschrijven of collection-data te raken. Matching, setvalidatie, allowlist en collection-countcontrole blijven expliciete stopvoorwaarden. |
| 2026-07 | Phase 7B-2F7 | Toon beschikbare geneste kaartdetails in de gedeelde Card Detail-presentatie en laat lege secties weg. | De JSONB-catalogus blijft uitbreidbaar zonder nieuwe kolommen; progressive disclosure houdt de detailweergave compact en collectiegegevens blijven gescheiden van kaartmetadata. |
| 2026-07 | Phase 7B-2F8 | Introduce a versioned local `pokemon_tcg_data` manifest batch flow as read-only orchestration around the single-set importer. | Local dataset validation needs reproducible set selection, pinned source metadata and machine-readable reporting, but matching and safety validation remain centralized in `import-set.ts`. The local source cannot write; broader catalog writes require a separate approved phase. |
| 2026-07 | Phase 7B-2F9A | Generate the local manifest automatically from a clean checkout at the exact pinned `PokemonTCG/pokemon-tcg-data` commit; use `sets/en.json` for the complete set list, but use each validated card-file length as manifest `expectedCards`. | The index total is an integrity signal (`indexedCardsTotal`, expected 20,219), while the readable card files are the actual import input (`receivedCardsTotal`, expected 20,324). Count differences are deterministic warnings; missing/unreadable files, invalid JSON and duplicate or missing IDs remain blocking. The complete catalog dry-run remains Phase 7B-2F9B. |
| 2026-07 | Phase 7B-2F9B | Bind a resumable local dry-run checkpoint to the exact source, pinned dataset, manifest bytes, ordered set selection and Supabase project identity; persist only sanitized status/totals atomically. | Long local runs must recover after interruption without silently mixing datasets, selections or Supabase projects. `passed` sets are safely skipped on resume; `pending`, `running` and `failed` sets are rerun. The checkpoint and report never contain secrets, payloads or subprocess output, and local dry-run database writes remain exactly zero. |
| 2026-07 | Phase 7C-2B | The first add-from-set flow creates exactly one owned Near Mint copy in the active collection. | This matches the existing INSERT policy, keeps the first write flow deterministic and avoids introducing condition, status and quantity choices before their UX and update semantics are designed. |
| 2026-07 | Phase 7C-2C1 | Collection quantity mutations are constrained at database level. | Frontend controls alone cannot protect data integrity. Updates may only change quantity by one, immutable card identity fields remain unchanged, and deletion represents the transition from one owned copy to zero. |
| 2026-07 | Phase 7C-2C2 | Set binders show card imagery only, with a subtle confirmed-presence marker. | Removing names, numbers, rarity and controls from each tile creates a cleaner digital-binder experience and keeps cards visually dominant. |
| 2026-07 | Phase 7C-2C2 | Card metadata and collection management belong in card detail, not in the binder grid. | Progressive disclosure keeps overview screens calm while preserving full management capability after opening a card. |
| 2026-07 | Card detail | Build the definitive card-detail experience as a shared component in a separate phase. | Sets, Collection and later Wishlist, Trade and Search need a consistent detail experience. PR 106 proved the functional flow, but the reusable component boundary deserves independent design and review. |
| 2026-07 | Images | Store external small and large image URLs initially; use thumbnails, detail loading and lazy loading. | This avoids unnecessary storage and keeps lists fast. Local caching can be reconsidered only if reliability or performance evidence requires it. |
| 2026-07 | Price data | Keep price synchronization separate from core card metadata. | Prices change much faster and have different source, refresh and history requirements than stable card identity and artwork metadata. |
| 2026-07 | Phase 7D-1B | Reuse the existing secured Collection and Wishlist mutation services from global Search and confirm every write through a bounded ownership read of the selected card. | Search must not duplicate business rules or trust optimistic client state. Success requires the expected server truth; if confirmation fails after a completed write, retry repeats only the read so quantity changes and other writes cannot be applied twice. |
| 2026-07 | Documentation | Use a charter, architecture principles, UX guidelines, current status, roadmap, decision log and AI working agreement as distinct sources of truth. | New chats, Codex tasks and future contributors need concise durable context without duplicating entire conversations or mixing current facts with historical decisions. |

## How to add decisions

Add a row when a phase changes architecture, schema, RLS/security, data invariants, external integrations, import behavior or a foundational product or UX flow. Do not record every small implementation detail.

When a decision is superseded, keep the original row and add a new row that states what changed and why.
